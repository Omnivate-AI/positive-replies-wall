### [Bug] DELETE redactions/highlights does not enforce `source = 'admin'` (PII regression vector)

**Severity:** High
**Priority:** P1
**Status:** Open
**Area:** `app/api/admin/redactions/route.ts`, `app/api/admin/highlights/route.ts`, `app/admin/dashboard.tsx`

**Problem**
Both DELETE handlers accept `{ id }` and delete the matching row by primary key alone. Neither checks the row's `source` column before deleting. The `source` enum on these tables (`prw_redactions.source` and `prw_highlights.source`) is the load-bearing signal that tells the system which rows are auto-managed (regenerated by ingest/classify) vs admin-curated.

The `docs/m10-admin-and-public-wall.md:121` and `:123` documentation explicitly claim:
> `DELETE /api/admin/redactions` … "Server-side check refuses if source != 'admin'."
> `DELETE /api/admin/highlights` … "Server-side check refuses if source != 'admin'."

The route handlers do not implement that check. The client-side dashboard guards the delete button with `r.source === "admin"` (`app/admin/dashboard.tsx:842`, `:782`) and `removeRedaction()`/`removeHighlight()` early-return when `target.source !== "admin"` (lines 253, 357), but client-side guards are not a security boundary — a hand-crafted `DELETE /api/admin/redactions { id: <auto_lead row id> }` succeeds.

**Impact**
The most damaging case is the `auto_lead` rows in `prw_redactions`. These are populated by `runIngest` from the matched lead row in the per-client `<client>_leads` table — they encode the prospect's `first_name`, `last_name`, `company_name`, and full `email` (see `trigger/lib/mappers.ts:152-180`, `redactionsFromLead`). They are the primary mask preventing prospect PII from leaking onto the public wall.

If those rows are deleted (whether by an attacker exploiting ticket 001's open admin API, or by an internal user calling the API directly, or — eventually — by a buggy admin UI flow), the deletion is **not idempotently restored**. The `auto_lead` redaction is only re-created when ingest runs again for that thread, which only happens if Smartlead returns a new message for that lead × campaign. The ingestion code's "skip if no new inbound" branch and the upsert's `ignoreDuplicates: true` semantics mean a routine daily run can leave the deletion permanent.

Result: a deleted auto_lead row → next ISR revalidate of `/` → prospect's name + company + email visible on the public wall (rendered in the EmailReplyCard's "From: …" header line and inside the body wherever those tokens appear).

The same hole exists for `auto_classifier` rows in both tables, but their blast radius is smaller (third-party names mentioned in body) — those rows also re-seed automatically when classify is re-run with the same `PROMPT_VERSION`.

**Evidence**
- `app/api/admin/redactions/route.ts:58-79` — DELETE handler:
  ```ts
  const { data, error } = await sb
    .from("prw_redactions")
    .delete()
    .eq("id", parsed.id)
    .select();
  ```
  No `.eq("source", "admin")` filter, no pre-fetch + check.
- `app/api/admin/highlights/route.ts:57-78` — identical pattern.
- `docs/m10-admin-and-public-wall.md:121, 123` — documented contract that the implementation violates.
- Client-only guards (not a security boundary): `app/admin/dashboard.tsx:243-253`, `:351-357`, `:782-790`, `:842-850`.
- The auto_lead redaction set is the PII firewall — see `trigger/lib/mappers.ts:152-180` and `lib/supabase-public.ts:317` (renderer adds the SDR allowlist on top, but the lead's own name/company/email comes from `prw_redactions`).
- Re-ingest behavior: `trigger/lib/ingest.ts:262-282` — auto_lead rows are upserted with `ignoreDuplicates: true` ON CONFLICT — i.e. once deleted, only re-created if the upsert tries to insert it again, which only happens when ingest reaches that thread.
- Linked: ticket 001 (no auth on these routes). Without auth, this bug is exploitable from the open internet; with auth, it's still a real internal risk because admins delete with one click.

**Expected behavior**
DELETE on `prw_redactions` or `prw_highlights` succeeds only when the row's `source = 'admin'`. Any other source returns 403 with a clear error and a `removed: 0` count.

The auto-source rows must remain immutable through the API surface, mirroring the documented contract and the dashboard's own UX rule (auto rows render without a delete button).

**Suggested fix**
In each DELETE handler, change the chained call to:

```ts
const { data, error } = await sb
  .from("prw_redactions")
  .delete()
  .eq("id", parsed.id)
  .eq("source", "admin")
  .select();
if (error) return NextResponse.json({ error: "db_error", message: error.message }, { status: 500 });
if (!data || data.length === 0) {
  // Either the row didn't exist or wasn't admin-source. Treat as 403.
  return NextResponse.json(
    { error: "forbidden", message: "Only admin-source rows can be deleted." },
    { status: 403 },
  );
}
return NextResponse.json({ ok: true, removed: data.length });
```

Apply the same to `app/api/admin/highlights/route.ts`. Add an integration test under `tests/integration/` that asserts a DELETE against an `auto_lead` redaction row returns 403 and the row is still present.

**Acceptance criteria**
- [ ] DELETE on a row with `source = 'auto_lead'` returns 403 and the row remains in the table.
- [ ] DELETE on a row with `source = 'auto_classifier'` returns 403 and the row remains.
- [ ] DELETE on a row with `source = 'admin'` returns 200 and the row is removed.
- [ ] An integration test exercises all three paths against the live `prw_redactions` and `prw_highlights` tables.
- [ ] The m10 documentation claim continues to hold (the implementation now matches the docstring).
